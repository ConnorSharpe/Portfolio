Progress Bars

This is how I built the most accurate progress bar that the Tech Academy has ever used. When I was a student, the progress bar only counted the actual course material which leaves between 2-4 weeks (depending on the particular bootcamp) for the mandatory live projects unaccounted for. The old model attempted to tell students how many days ahead/behind they were and then the student had to mentally account for the amount of time missing from the calculation. I was tasked with preserving the days ahead/behind while making the whole process more accurate. All of the courses at the Tech Academy are weighted by the days it takes on average to complete given an assumed 8 hour study day. The issue here is that our largest course page-wise, Computer and Technology Basics, contains over 800 pages - but only takes 3 days to complete because it is not coding intensive. While C# and .Net Framework Course contains only 300 pages but takes about two weeks to complete because of the amount of small coding projects contained within. To top it all off, not all boot camps contain the same classes or completion time (between 10-16 weeks full-time). I reorganized the way that bootcamps were stored in the dB. Since 1 course can exist in some or all camps, I created the LMSCourseOrder table. This is just an object to link a course to a bootcamp. A boot camp itself is just the collection of courseOrders with that boot camp id.

To include the previously unincluded 2 week live projects, I created a live Project course containing only the instructions needed to access the team story boards (about 6 pages) and weighted it at 2 weeks. Boot camps with multiple live projects could now reuse the same live project course in different course orders to accommodate the lengthier camps. Next, I had to find the course progress that the student had completed - a simple matter of dividing the page number they were on by the total pages in the course and rounding that number to a percent. To calculate where the student should be, I would need to grab the user start date and the weeks they purchased to compare it to the previously calculated Boot Camp progress. Drill Lock

A common complaint that instructors had in the past was that there was nothing in place to ensure that all of the coding assignments that students submit are passed successfully. Students either need to go to their inbox on their own volition to see if they have any failed submissions OR they need to fail enough assignments to warrant being contacted by the instructors. I needed to place restrictions on further course material while allowing the user to access prior pages (there must be something they didn't understand in order to fail the task). I started out by changing the existing boolean grading attribute to a nullable "bool?" called Passed - this means that the default value went from false to null. If an assignment has a null grade, the instructors know that it has not been graded yet.

Then I added a boolean attribute called FailedAssignmentLock to the CurrentCourse object that will become true when an instructor fails an assignment by that student:

Students can only further their position by clicking the next button on their current position page. So inside of that Next controller method, I added an If statement that checks if they are trying to advance their position while locked (that way they still have access to revious material in case they need a refresher).

I built a super simple method and view to be shown if the student's parameters fall inside that If statement:

In order to free themselves from the lock, students need to access their inboxes and resubmit any failed drills. This act would change the assignments grade from failed to null - alerting the instructors to re-assess the method. Resubmission hits this block of code which checks the dB to see if there are any other failed assignments by the student, if not then DrillLock = false. Thus they are allowed to continue learning until an instructor fails another assignment:

Even if they haven't completed the assignment to the instructor's standards, when they resubmit, they can continue on in the system until the instructors regrade the assignment with a failure. This was done so that the students won't need to wait around until an instructor verifies the assignment as passing.
